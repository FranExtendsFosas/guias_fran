Bases de Datos Relacionales

La guía definitiva para entender, diseñar y desplegar bases de datos relacionales en entornos reales. Desde los fundamentos matemáticos hasta la operación en producción.

🎯 De Cero a Producción 📄 Documento Único 🎨 One Dark Vivid

01¿Qué es una Base de Datos Relacional?

Una base de datos relacional es un sistema de almacenamiento de información que organiza los datos en estructuras tabulares (filas y columnas) y, lo más importante, establece relaciones formales entre esas tablas. No es simplemente "hojas de cálculo conectadas": es una implementación práctica de la teoría matemática de conjuntos y la lógica de predicados de primer orden.

💡
Idea Central

La revolución de Edgar F. Codd (1970) fue proponer que los datos deben organizarse según relaciones matemáticas, no según estructuras físicas (como los archivos jerárquicos de la época). Esto separó la representación lógica de los datos de su almacenamiento físico.

El corazón del modelo relacional es el concepto de relación. En matemáticas, una relación es un subconjunto del producto cartesiano de dominios. En términos prácticos: una tabla donde cada fila es única y el orden de las filas no importa.

¿Por qué "Relacional"?

El nombre proviene de relation (relación), no de "relacionar tablas". Aunque las relaciones entre tablas son cruciales, el término original describe la estructura matemática subyacente. Cada tabla es una relación; cada fila es una tupla; cada columna es un atributo.

El RDBMS

Un Sistema Gestor de Bases de Datos Relacionales (RDBMS) es el software que implementa el modelo relacional. No solo almacena datos: garantiza reglas, optimiza consultas, gestiona concurrencia y recuperación. Es un sistema operativo especializado para datos estructurados.

Componente Función Ejemplo en Producción
Motor de Almacenamiento Gestiona cómo se escriben y leen los datos en disco InnoDB (MySQL), Heap (PostgreSQL)
Optimizador de Consultas Decide el plan de ejecución más eficiente Cost-based optimizer (PostgreSQL)
Gestor de Transacciones Garantiza ACID y control de concurrencia MVCC (Multi-Version Concurrency Control)
Catálogo de Sistema Metadatos: tablas, columnas, índices, permisos information_schema, pg_catalog

02El Modelo Relacional: Fundamentos Matemáticos

Entender las bases matemáticas te permite tomar decisiones de diseño informadas, no solo seguir recetas. El modelo relacional descansa sobre tres pilares:

1. Estructura: Relaciones y Tuplas

Una relación R sobre dominios D1, D2, ..., Dn es un subconjunto del producto cartesiano D1 × D2 × ... × Dn. Cada elemento de este subconjunto es una tupla (d1, d2, ..., dn) donde di ∈ Di.

En la práctica: una tabla donde cada columna tiene un dominio (tipo de dato) y cada fila es una combinación válida de valores de esos dominios.

⚠️
Restricción Fundamental

En una relación matemática pura, no hay tuplas duplicadas. El RDBMS impone esto mediante claves primarias. Sin una PK, técnicamente no es una relación válida.

2. Integridad: Reglas del Mundo Real

La integridad de entidad dice que toda tupla debe ser identificable (PK no nula y única). La integridad referencial dice que las referencias entre relaciones deben ser válidas (las FK deben apuntar a PK existentes o ser nulas).

3. Manipulación: Álgebra Relacional

Codd definió un conjunto de operaciones formales para manipular relaciones: selección (σ), proyección (π), unión (∪), diferencia (−), producto cartesiano (×) y join (⋈). SQL es una implementación práctica (y superconjunto) de este álgebra.

03Tablas, Filas y Columnas: El Lenguaje del RDBMS

Aunque usamos "tabla" coloquialmente, en un RDBMS existen distinciones importantes:

Tabla (Table / Relation)

Estructura con nombre que define un conjunto de tuplas del mismo tipo. En SQL: CREATE TABLE. Una tabla tiene:

  • Nombre único en el esquema
  • Columnas con tipos de datos y restricciones
  • Filas (tuplas) que cumplen el esquema
  • Restricciones (constraints) que limitan los valores posibles

Columna (Column / Attribute)

Define un dominio. El tipo de dato no es decorativo: determina el almacenamiento físico, las operaciones válidas y las optimizaciones posibles.

Tipo Conceptual Tipos Comunes Consideraciones de Producción
Numéricos exactos INTEGER, BIGINT, NUMERIC NUMERIC es preciso pero más lento que INTEGER. Usa BIGINT para IDs (agotar INTEGER es un incidente real).
Numéricos aprox. REAL, DOUBLE PRECISION, FLOAT Nunca para dinero. Redondeo impredecible. Prohibido en contabilidad.
Texto VARCHAR(n), TEXT, CHAR(n) TEXT sin límite es preferible en PostgreSQL. VARCHAR solo si hay restricción de negocio real.
Temporal TIMESTAMP, DATE, TIMESTAMPTZ Siempre TIMESTAMPTZ (con zona horaria). Almacenar en UTC, mostrar en local.
Booleano BOOLEAN Algunos motores usan BIT o TINYINT(1). PostgreSQL tiene booleano nativo.
Binarios BYTEA, BLOB Para archivos pequeños. Para grandes, usa almacenamiento de objetos (S3) y guarda la referencia.

Fila (Row / Tuple)

Una instancia concreta de la relación. Las filas no tienen orden intrínseco. Si necesitas orden, usa ORDER BY. El RDBMS puede devolverlas en cualquier secuencia física.

🛑
Anti-patrón Común

Confiar en el "orden natural" de las filas. No existe. El orden físico depende del plan de acceso, índices, vacuums, reordenamientos. Siempre explicita el orden lógico que necesitas.

04Claves: Identidad y Referencia

Las claves son el mecanismo que transforma una tabla en una relación matemática válida y que permite navegar entre relaciones.

Clave Primaria (Primary Key - PK)

Identifica unívocamente cada fila. Propiedades:

  • Unicidad: No puede repetirse en toda la tabla
  • Irreductibilidad: No debe contener atributos superfluos (si es compuesta)
  • No nulidad: Ninguna parte de la PK puede ser NULL
  • Inmutabilidad ideal: No debe cambiar (cambiar una PK es costoso y arriesgado)
🎯
Estrategia de Producción: Surrogate Keys

Usa claves surrogadas (artificiales) como autoincrementales o UUIDs como PK, en lugar de datos de negocio (DNI, email). Los datos de negocio cambian, las surrogadas no. Esto desacopla la identidad física de la identidad de negocio.

Clave Foránea (Foreign Key - FK)

Establece una relación de integridad referencial entre dos tablas. Una FK en la tabla hija referencia la PK de la tabla padre. El RDBMS impide que existan "huérfanos".

┌─────────────────────────────────────────────────────────────────────────────────────┐ │ Tabla: CLIENTES Tabla: PEDIDOS │ │ │ │ ┌─────────────┬──────────┐ ┌─────────────┬─────────────┬──────────┐ │ │ cliente_id │ nombre │ │ pedido_id │ cliente_id │ total │ │ ├─────────────┼──────────┤ ├─────────────┼─────────────┼──────────┤ │ │ 101 │ Acme Corp│◄───────────────────│ 5001 │ 101 │ 1200.00 │ │ │ 102 │ Globex │ FK constraint │ 5002 │ 101 │ 850.00 │ │ │ 103 │ Initech │ │ 5003 │ 102 │ 2300.00 │ │ └─────────────┴──────────┘ └─────────────┴─────────────┴──────────┘ │ ↑ ↑ │ PK (única, no nula) FK (referencia PK) │ │ -- Si intentas borrar el cliente 101, el RDBMS bloquea la operación │ -- porque existen pedidos dependientes (a menos que configures ON DELETE CASCADE) └─────────────────────────────────────────────────────────────────────────────────────┘

Clave Única (Unique Key - UK)

Garantiza unicidad sin ser identificador primario. Útil para datos de negocio que deben ser únicos (email, SKU, ISBN) pero que no usas como referencia en otras tablas.

Clave Compuesta

PK formada por múltiples columnas. Útil en tablas de unión (junction tables) para relaciones N:M. Ejemplo: PK compuesta por (pedido_id, producto_id) en una tabla de líneas de pedido.

⚡
Impacto en Índices

Toda PK y UK crea automáticamente un índice único en el RDBMS. Esto acelera las búsquedas por esa columna, pero ralentiza las inserciones/actualizaciones porque el índice debe mantenerse. Diseña tus claves pensando en el balance lectura/escritura.

05Tipos de Relaciones entre Tablas

La cardinalidad define cuántas instancias de una entidad se asocian con instancias de otra. Es fundamental para modelar correctamente el dominio.

1:1 (Uno a Uno)

Una fila en A corresponde a exactamente una fila en B. Se implementa colocando la PK de A como PK y FK en B (o viceversa). Útil para:

  • Separar datos sensibles (tabla de contraseñas/hash separada)
  • Datos opcionales y voluminosos (perfil extendido de usuario)
  • Particionamiento lógico por acceso (datos frecuentes vs. raros)

1:N (Uno a Muchos)

La relación más común. Una fila en A tiene muchas filas en B. Se implementa con una FK en B que referencia A. Ejemplo: un cliente tiene muchos pedidos.

N:M (Muchos a Muchos)

No se implementa directamente. Se descompone en dos relaciones 1:N mediante una tabla de unión (junction/associative table). Esta tabla contiene FKs a ambas entidades; a menudo su PK es la composición de ambas FKs.

✅
Regla de Oro del Modelado

Siempre pregunta: "¿Esta entidad tiene vida propia independiente?" Si una "categoría" solo existe dentro de un "producto", quizás sea un atributo enum, no una tabla. Si puede existir sin el producto (y otros productos la usan), es una entidad con tabla propia y relación FK.

06Normalización: Eliminando Anomalías

La normalización es el proceso sistemático de organizar columnas y tablas para minimizar la redundancia y eliminar anomalías de inserción, actualización y borrado. Codd definió las Formas Normales (FN) como objetivos sucesivos.

Primera Forma Normal (1FN)

Regla: Todos los atributos contienen valores atómicos (indivisibles) y no hay grupos repetitivos.

Anomalía que corrige: Listas dentro de una celda (emails múltiples en un campo, tags separados por comas).

Solución: Crear tabla separada para el grupo repetitivo con FK a la tabla principal.

Segunda Forma Normal (2FN)

Regla: Está en 1FN y todos los atributos no-clave dependen funcionalmente de toda la PK (no de parte de ella).

Aplicable a: Tablas con PK compuesta.

Anomalía: Si (pedido_id, producto_id) es PK, y guardas el nombre_cliente en esa tabla, depende solo de pedido_id, no de toda la PK. Mueve nombre_cliente a la tabla de pedidos.

Tercera Forma Normal (3FN)

Regla: Está en 2FN y no existen dependencias transitivas (atributos no-clave que dependen de otros atributos no-clave).

Ejemplo clásico: Tabla de empleados con (id, nombre, departamento_id, nombre_departamento, ubicacion_departamento). El nombre y ubicación del departamento dependen de departamento_id, no directamente del id del empleado. Solución: tabla separada DEPARTAMENTOS.

Forma Normal de Boyce-Codd (BCNF)

Variante más estricta de 3FN. Para cada dependencia funcional X → Y, X debe ser una superclave. Más exigente que 3FN pero a veces requiere descomposiciones que pierden dependencias funcionales.

🎓
Perspectiva Práctica

En producción, la mayoría de los diseños saludables alcanzan 3FN o BCNF. Las formas superiores (4FN, 5FN) tratan dependencias multivaluadas y join, pero son raramente necesarias en aplicaciones comerciales. Normaliza hasta eliminar la redundancia obvia, luego desnormaliza selectivamente si hay problemas de rendimiento medidos.

Desnormalización Controlada

En sistemas de lectura intensiva (reporting, analytics), a veces se introducen redundancias calculadas (campos derivados, tablas agregadas) para evitar joins costosos. Esto es una desnormalización deliberada, no un error de diseño. La clave:

  • Documentar que es intencional
  • Mantener la fuente de verdad normalizada
  • Actualizar los datos derivados de forma transaccional o eventualmente consistente
  • Medir antes: no optimices prematuramente

07Integridad Referencial y Constraints

La integridad no es opcional en producción. Los constraints (restricciones) son reglas declarativas que el RDBMS garantiza, independientemente de qué aplicación acceda a los datos.

Tipos de Constraints

Constraint Propósito Ejemplo de Producción
PRIMARY KEY Unicidad + No NULL Toda tabla debe tener una. Es el ancla de tu modelo.
FOREIGN KEY Integridad referencial Siempre que haya relación lógica. Sin FK, los datos se corrompen.
UNIQUE Unicidad sin ser PK Emails de usuario, números de factura, slugs de URL.
NOT NULL Obligatoriedad Campos críticos: fecha de creación, estado, monto.
CHECK Validación de dominio CHECK (precio > 0), CHECK (estado IN ('activo', 'inactivo'))
DEFAULT Valor por omisión created_at DEFAULT NOW(), estado DEFAULT 'pendiente'

Acciones Referenciales (ON DELETE / ON UPDATE)

Cuando la fila padre cambia, ¿qué pasa con las hijas?

  • NO ACTION / RESTRICT: Bloquea la operación si hay hijas (por defecto, el más seguro)
  • CASCADE: Propaga el cambio (borra/actualiza hijas automáticamente). Úsalo con precaución: un borrado accidental puede eliminar cascadas masivas.
  • SET NULL: Pone la FK a NULL (solo si la columna lo permite)
  • SET DEFAULT: Asigna un valor por defecto
🔥
Error Crítico en Producción

Deshabilitar las claves foráneas "por rendimiento". Esto destruye la integridad de datos. Si una aplicación escribe datos inconsistentes (por bugs, race conditions, o escrituras manuales), recuperar la consistencia es extremadamente difícil. Las FK son la última línea de defensa de tu modelo de datos.

08Transacciones ACID: Garantías de Consistencia

Una transacción es una unidad lógica de trabajo que agrupa múltiples operaciones. El RDBMS garantiza que, independientemente de fallos, concurrencia o cortes de energía, las transacciones cumplen ACID:

A - Atomicidad (Atomicity)

Todo o nada. Las operaciones de la transacción se completan íntegramente, o ninguna tiene efecto. Si falla la mitad, el RDBMS hace rollback automático al estado previo usando el Write-Ahead Log (WAL).

C - Consistencia (Consistency)

Cada transacción lleva la base de datos de un estado válido a otro estado válido. Todas las reglas (constraints, triggers, cascadas) se cumplen al finalizar.

I - Aislamiento (Isolation)

Las transacciones concurrentes no se interfieren entre sí. El resultado es el mismo que si se ejecutaran secuencialmente. Se implementa mediante locking y MVCC (Multi-Version Concurrency Control).

D - Durabilidad (Durability)

Una vez confirmada (COMMIT), la transacción sobrevive a cualquier fallo. Los datos se escriben en almacenamiento persistente (disco) antes de reportar éxito al cliente.

Niveles de Aislamiento

El estándar SQL define 4 niveles. Cuanto más alto, más consistencia pero menos concurrencia:

Nivel Fenómenos Permitidos Uso Típico
READ UNCOMMITTED Dirty reads, non-repeatable, phantom Casi nunca. Solo reporting donde la exactitud no importa.
READ COMMITTED Non-repeatable reads, phantom Default en muchos motores. Buen balance para OLTP.
REPEATABLE READ Phantom reads (en algunos motores) Default en MySQL (InnoDB). Consistente para la transacción.
SERIALIZABLE Ninguno Operaciones críticas (transferencias bancarias). Máximo overhead.
🔄
MVCC: La Magia Moderna

PostgreSQL y Oracle usan MVCC en lugar de locks pesados. Cada transacción ve una "fotografía" consistente de la base de datos al momento de iniciar. Las escrituras crean nuevas versiones sin bloquear lecturas. Esto permite alta concurrencia sin deadlocks constantes.

09Índices y Estructuras de Acceso

Un índice es una estructura de datos auxiliar que acelera la recuperación de filas a costa de espacio adicional y lentitud en escrituras. Sin índices, el RDBMS debe hacer Sequential Scan (leer toda la tabla).

B-Tree: El Estándar Universal

La estructura dominante. Un B-Tree (Balanced Tree) mantiene los datos ordenados en una estructura de árbol auto-balanceada. Cada nodo tiene múltiples hijos. Garantiza:

  • Búsqueda, inserción y borrado en O(log n)
  • Ordenación implícita (útil para ORDER BY y rangos)
  • Balanceo automático: todas las hojas están a la misma profundidad
-- Representación conceptual de un B-Tree para índice en "email" [m@mail.com] / \ [a@mail.com] [r@mail.com] / \ / \ [a@..][j@..] [l@..][m@..] [r@..][s@..] [z@..] -- Las hojas contienen punteros a las filas físicas (TIDs/ROWIDs) -- La profundidad es ~logarítmica respecto al número de filas

Tipos de Índices

Tipo Mejor Para Notas de Producción
B-Tree Igualdad, rangos, ordenación Default en la mayoría de motores. Útil para >, <, =, BETWEEN, LIKE 'abc%'.
Hash Igualdad exacta (=) PostgreSQL los crea internamente para joins hash. No soporta rangos.
GiST / GIN Datos complejos, arrays, full-text PostgreSQL. GIN es excelente para arrays y búsqueda de texto.
Partial Subconjuntos frecuentes WHERE activo = true. Índice más pequeño y rápido.
Composite Consultas multi-columna Orden importa: (apellido, nombre) acelera WHERE apellido = X pero no WHERE nombre = X solo.
Covering Index-only scans Incluye columnas adicionales para evitar acceso a la tabla heap.
⚠️
Anti-patrón: Índice en Todo

Cada índice ralentiza INSERT, UPDATE y DELETE porque debe mantenerse actualizado. Además, consume espacio en disco y memoria (shared buffers). Regla: indexa las columnas en WHERE, JOIN y ORDER BY de consultas frecuentes. Usa EXPLAIN ANALYZE para verificar que el índice se usa realmente.

10Motores RDBMS: Elegir tu Arma

No todos los RDBMS son iguales. La elección afecta arquitectura, operación y coste a largo plazo.

Motor Filosofía Fortalezas Casos de Uso
PostgreSQL Correctitud, extensibilidad, estándares Tipos avanzados, JSONB, extensiones (PostGIS), MVCC robusto, comunidad activa Aplicaciones complejas, GIS, data warehousing ligero, startups a enterprise
MySQL (InnoDB) Simplicidad, velocidad, amplia adopción Fácil de operar, replicación madura, ecosistema LAMP masivo Web apps, CMS, e-commerce, cuando la operación simple prima

Criterios de Selección para Producción

  1. Equipo: ¿Qué conocen bien? La curva de aprendizaje opera 24/7.
  2. Ecosistema: Drivers, ORMs, herramientas de backup/monitoring.
  3. Escalabilidad vertical vs. horizontal: ¿Necesitarás sharding? PostgreSQL y MySQL manejan TBs sin sharding; más allá, requiere arquitectura.
  4. Coste total: Licencias vs. tiempo de operación vs. hardware.
  5. Features específicas: Full-text search, geoespacial, JSON nativo, particionamiento.

11Almacenamiento Físico y Arquitectura Interna

Comprender cómo se almacenan los datos físicamente te ayuda a diagnosticar problemas de rendimiento y a dimensionar infraestructura.

Páginas (Pages / Blocks)

El RDBMS lee y escribe en unidades de páginas (típicamente 8KB en PostgreSQL, 16KB en InnoDB). Una fila nunca se divide entre páginas. Las páginas se almacenan en:

  • Shared Buffers: Cache en memoria RAM de páginas recientes
  • Tablespace files: Archivos en disco donde residen permanentemente

Heap vs. Índice

La tabla heap es el almacenamiento principal desordenado de filas. Los índices contienen valores ordenados + punteros (TID/ROWID) a la ubicación en el heap. Una consulta por índice hace:

  1. Recorre el B-Tree del índice para encontrar las entradas coincidentes
  2. Obtiene los punteros a las páginas del heap
  3. Accede al heap para recuperar las columnas no presentes en el índice (excepto en Index-Only Scan)

TOAST (The Oversized-Attribute Storage Technique)

En PostgreSQL, las filas no pueden exceder ~8KB. Los valores grandes (TEXT largo, JSONB grande) se comprimen y almacenan en tablas TOAST separadas, con un puntero en la fila principal. Esto es transparente pero afecta el rendimiento de lectura de campos grandes.

WAL (Write-Ahead Logging)

Toda modificación se escribe primero en un log secuencial (WAL/redo log) antes de tocar los datos. Esto permite:

  • Recuperación tras crash: Reproduce el WAL para restaurar consistencia
  • Replicación: El standby aplica el WAL del primario
  • Point-in-Time Recovery: Restaurar a cualquier momento usando backups base + WAL archivado

12Concurrencia, Locking y Deadlocks

En producción, múltiples usuarios y procesos acceden simultáneamente. El RDBMS debe coordinar esto sin corrupción ni bloqueos eternos.

Niveles de Locking

Nivel Qué Bloquea Ejemplo
Row-level Solo la fila afectada InnoDB, PostgreSQL. Máxima concurrencia.
Page-level Página de 8KB (múltiples filas) SQL Server en ciertos casos. Menos concurrente.
Table-level Toda la tabla MyISAM, DDL statements (ALTER TABLE). Bloquea todo.

Tipos de Locks

  • Shared Lock (S): Para lecturas. Múltiples transacciones pueden tener S locks simultáneos.
  • Exclusive Lock (X): Para escrituras. Bloquea a otros S y X.
  • Intent Locks: Indican la intención de bloquear a nivel inferior (evita conflictos entre niveles)
  • Serializable: Bloqueos de predicado para evitar phantoms

Deadlocks

Ocurren cuando la transacción A espera un recurso de B, y B espera uno de A. El RDBMS detecta ciclos y mata una transacción (rollback) para desbloquear. En producción:

  • Accede a los recursos siempre en el mismo orden en toda la aplicación
  • Mantén las transacciones cortas
  • Usa timeouts de lock wait
  • Captura los errores de deadlock en la aplicación y reintenta

13Replicación y Alta Disponibilidad

En producción, un solo servidor es un single point of failure. La replicación distribuye datos y carga.

Replicación Física (Streaming Replication)

El standby recibe el WAL binario del primario y lo aplica byte por byte. Es idéntico al primario. PostgreSQL y MySQL lo soportan nativamente.

  • Síncrona: El primario espera confirmación del standby antes de reportar COMMIT. Cero pérdida de datos, pero latencia.
  • Asíncrona: El primario no espera. Mínima latencia, pero posible lag (retraso) y pérdida de datos si el primario falla antes de que el standby reciba el WAL.

Replicación Lógica

Replica a nivel de fila/columna, no de bytes. Permite:

  • Replicar solo tablas específicas
  • Replicar entre versiones diferentes del motor
  • Replicar a bases de datos con esquemas diferentes

Topologías

Topología Descripción Cuándo Usarla
Primario-Standby Uno activo, uno o más pasivos Alta disponibilidad básica, failover manual o automático
Cascada Standby alimenta otro standby Reducir carga del primario, geografía distribuida
Multi-primario Múltiples nodos aceptan escrituras MySQL Group Replication, Galera. Conflictos complejos; úsalo si entiendes las limitaciones.
🏗️
Arquitectura Típica de Producción

Primario (escrituras) + 2+ Réplicas de Lectura (reporting, análisis) + Failover Automático (Patroni, repmgr, Orchestrator). Las réplicas alivian el primario de consultas pesadas y sirven como respaldo inmediato.

14Backups y Recuperación ante Desastres

Los backups no son opcionales. Un sistema sin backups verificados es un sistema que eventualmente perderá datos.

Tipos de Backup

  • Físico (Base Backup): Copia de los archivos de datos del disco. Rápido de restaurar. Requiere consistencia (snapshot o modo backup).
  • Lógico (SQL Dump): pg_dump, mysqldump. Archivo SQL con comandos de recreación. Portable entre versiones, pero lento para bases grandes.
  • Incremental (WAL Archiving): Archiva los WAL continuamente. Permite Point-in-Time Recovery (PITR) a cualquier momento entre backups base.

Estrategia 3-2-1

Estándar de la industria para backups:

  • 3 copias de los datos
  • En 2 tipos de medio diferentes (disco local, nube)
  • 1 copia fuera del sitio (offsite/cloud)

Verificación de Backups

Un backup no verificado no existe. Debes:

  1. Restaurar periódicamente en un entorno aislado
  2. Ejecutar CHECKSUM o validación de integridad
  3. Medir el tiempo de restauración (RTO - Recovery Time Objective)
  4. Definir cuánta pérdida de datos es aceptable (RPO - Recovery Point Objective)
🚨
Incidente Clásico

"Hacíamos backups diarios, pero nunca los probamos. Cuando el disco falló, descubrimos que los dumps estaban corruptos desde hacía 3 meses." Restaura tus backups. Automatiza la verificación. Documenta el procedimiento de recuperación.

15Seguridad en Bases de Datos

La base de datos es el tesoro. Su protección es multicapa.

1. Autenticación y Autorización

  • Usuarios con privilegios mínimos: La app no necesita ser SUPERUSER. Crea roles específicos: app_read, app_write, app_admin.
  • Contraseñas fuertes + SSL: Siempre conexiones cifradas en producción. Nunca passwords en texto plano en logs o repos.
  • Autenticación externa: LDAP, Kerberos, o certificados de cliente cuando sea posible.

2. Cifrado

  • En tránsito (TLS/SSL): Obligatorio. Configura sslmode=require o equivalente.
  • En reposo (TDE): Transparent Data Encryption cifra los archivos de datos. Protege contra robo físico de discos.
  • A nivel de columna: Cifra datos sensibles (tarjetas, documentos) antes de insertar, o usa funciones de cifrado del motor (pgcrypto).

3. Auditoría

Habilita logging de:

  • Conexiones y desconexiones
  • Consultas DDL (CREATE, ALTER, DROP)
  • Intentos de acceso fallidos
  • Cambios en tablas sensibles (triggers de auditoría o extensiones como pgaudit)

4. Protección contra Inyección SQL

No es solo "escapar strings". Es arquitectura:

  • Usa parametrización (prepared statements) SIEMPRE
  • Nunca concatenes input de usuario en queries
  • Usa ORMs correctamente (evita raw() o query() con strings dinámicos)
  • Principio de mínimo privilegio: la cuenta de la app no puede ejecutar DROP TABLE

16Monitoreo y Observabilidad

No puedes operar lo que no puedes ver. El monitoreo debe ser proactivo, no reactivo.

Métricas Críticas (The Four Golden Signals)

Señal Métrica de BD Umbral de Alerta
Latencia Tiempo promedio de consulta (avg query time) > 100ms para OLTP, > 5s para reporting
Tráfico Conexiones activas, transacciones por segundo (TPS) > 80% del max_connections
Errores Deadlocks, check constraint failures, connection errors Cualquier deadlock en producción merece investigación
Saturación CPU, IOPS de disco, uso de memoria (shared buffers), WAL lag CPU > 70% sostenido, disco > 80% capacidad

Herramientas Esenciales

  • pg_stat_statements / Performance Schema: Identifica las queries más lentas y frecuentes
  • EXPLAIN (ANALYZE, BUFFERS): El plan de ejecución. Tu herramienta #1 para optimización
  • Logs lentos: Configura log_min_duration_statement para capturar queries > N ms
  • Dashboards: Grafana + Prometheus/InfluxDB para visualización histórica
  • Alerting: PagerDuty/Opsgenie para incidentes críticos
📊
Query de Diagnóstico Rápido (PostgreSQL)
SELECT query, calls, mean_exec_time, rows
FROM pg_stat_statements
ORDER BY mean_exec_time * calls DESC
LIMIT 10;
-- Muestra las queries que más tiempo consumen en total
-- Prioriza optimizar las que aparecen primero

17Checklist: De Cero a Producción

Esta lista condensa todo lo anterior en acciones verificables. Imprímela, táchala, no despliegues sin ella.

Fase 1: Diseño y Modelado

  • Todas las tablas tienen PK (preferiblemente surrogate: BIGSERIAL/UUID)
  • Las relaciones 1:N usan FK con acciones referenciales definidas
  • Las relaciones N:M usan tabla de unión con PK compuesta
  • El modelo está al menos en 3FN (salvo desnormalización documentada)
  • Los tipos de datos son apropiados (TIMESTAMPTZ, no FLOAT para dinero)
  • Todos los campos de negocio tienen constraints (CHECK, NOT NULL, UNIQUE)

Fase 2: Índices y Rendimiento

  • Índices en todas las FK (el RDBMS no los crea automáticamente)
  • Índices en columnas de búsqueda frecuente (WHERE, JOIN, ORDER BY)
  • Se verificó con EXPLAIN ANALYZE que los índices se usan
  • Se eliminaron índices redundantes o no utilizados
  • Se consideraron índices parciales para subconjuntos frecuentes

Fase 3: Seguridad

  • La cuenta de la aplicación tiene privilegios mínimos
  • Las conexiones usan SSL/TLS obligatorio
  • Las contraseñas no están en repositorios (usa secret management)
  • Datos sensibles están cifrados (column-level o TDE)
  • Logging de auditoría habilitado para DDL y accesos fallidos

Fase 4: Operación

  • Backups físicos + lógicos configurados y verificados
  • WAL archiving habilitado para PITR
  • Replicación configurada (al menos 1 standby)
  • Monitoreo de las 4 golden signals implementado
  • Alertas configuradas para: lag de réplica, deadlocks, espacio en disco, conexiones
  • Procedimiento de failover documentado y probado
  • Runbook de recuperación ante desastres actualizado

Fase 5: Mantenimiento Continuo

  • Vacuum/Analyze programados (autovacuum en PostgreSQL)
  • Reindexación periódica de índices fragmentados
  • Revisión mensual de queries lentas en pg_stat_statements
  • Actualizaciones de seguridad del motor planificadas
  • Test de restauración de backups trimestral
🚀
Listo para Producción

Si has completado esta checklist, tienes una base de datos relacional sólida, segura y operable. Recuerda: la base de datos es un sistema vivo. El diseño inicial es el 20%; el 80% es operación continua, monitoreo y evolución controlada.
